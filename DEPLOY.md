# Cloud Deployment — Public Multi-Tenant Site

Run InstaContentEngine as a **public site**: people register, enter their **own**
API keys in ⚙️ Settings, and generate + publish under their own account. The same
backend also does **24/7 scheduled publishing** (even when everyone's PC is off),
which the desktop app can't (Instagram has no native "publish at time X").

This guide uses **Render** (persistent Postgres + a disk + always-on web service).
Railway / Fly.io work the same way with the included `Dockerfile`.

## What you need

- This repo on GitHub.
- Nothing else up front — **users bring their own keys**. There are no
  platform-wide API keys to configure. (Each user needs their own OpenRouter key,
  and, to publish, an Instagram Graph token + imgbb key; for X, their own paid
  X API keys. They enter these in the site's ⚙️ Settings page.)

## Steps (Render)

1. Push to GitHub.
2. Render dashboard → **New → Blueprint** → select this repo. Render reads
   `render.yaml` and creates a **web service + Postgres + a 5 GB disk**.
3. In the service **Environment** tab set the `sync:false` vars:
   - `ADMIN_EMAILS` — your email (comma-separated for several). These accounts get
     access to DB **Backup/Restore**; on a cloud deploy nobody is admin otherwise.
   - `PUBLIC_BASE_URL` — only for publishing **Reels** (Instagram fetches the video
     by public URL). After the first deploy set it to `https://<your-app>.onrender.com`.
   - `SECRET_KEY` is **generated by Render automatically** — leave it.
4. Deploy. When live, open `https://<your-app>.onrender.com/health` → `{"status":"ok"}`,
   then `https://<your-app>.onrender.com/` → you'll see the **login / register** screen.

`DATABASE_URL` is injected from the Postgres addon and normalized to the async
driver (asyncpg) on startup; APScheduler stores its jobs in the same Postgres so
scheduled publishes survive restarts.

## ⚠️ SECRET_KEY is permanent

`SECRET_KEY` signs every login token **and** derives the key that encrypts users'
stored API keys. **Never change it after go-live** — rotating it logs everyone out
**and makes every stored credential undecryptable**. Render generates a strong,
stable value once; leave it alone.

## Using it (as a user)

1. Open the site → **Register** (email + password).
2. **⚙️ Settings** → paste your keys (OpenRouter for generation; Instagram token +
   user id + imgbb for IG publishing; X keys for X). They're stored encrypted and
   only ever used for your own posts.
3. Generate a post, hit **📅 Schedule**, pick a time. The cloud backend publishes
   at the scheduled moment with **your** keys — your PC can be off.

## Local desktop vs Cloud

| | Local (`.pyw` / `.exe`) | Cloud (Render) |
|---|---|---|
| Users | one (you), keys from `.env` | many, each registers + brings own keys |
| Login | none | email + password |
| Generate / edit / export | ✅ | ✅ |
| Scheduled publish, **PC off** | ❌ | ✅ |
| DB / storage | sqlite + local `uploads/` | Postgres + persistent disk |
| Cost | free | ~$7/mo (Render starter) + Postgres/disk |

## Notes

- **Persistent disk is required.** Generated slides, raw backgrounds and reel MP4s
  live under `/app/uploads`. The `render.yaml` disk keeps them across redeploys;
  without it they vanish on every deploy (breaks overlay-edit and image serving).
  Having a disk makes the service single-instance with a brief restart on deploy.
- The free Render tier **sleeps** and would miss scheduled jobs — use **starter**.
- **Backup/Restore** in the UI is limited to `ADMIN_EMAILS`. You can also rely on
  Render's automated Postgres backups (the disk holds only media files).
- imgbb images and reel videos are public-by-URL (unguessable) — fine, the content
  becomes public on Instagram anyway.
- Each user rotates their **own** Instagram token before it expires (~60 days).
- This is a "bring your own keys" model. OAuth "Connect Instagram/X" (so users
  don't paste raw tokens) needs Meta App Review and is a separate future step.
